# Marti — MVP кабинета поставщика

Рабочий PWA‑frontend для локальных поставщиков: поставщик находит выполненную поставку, получает подтверждение покупателя, проверяет комплектность и прибыльность, подписывает demo‑ЭЦП и контролирует оплату и регрессный риск до закрытия сделки.

## Запуск

Требование: установленный Node.js 20.16+ (или 22.3+ для `pdf-parse`).

### Самый простой способ

```bash
npm run demo
```

Команда при первом запуске сама установит зависимости, найдёт свободный порт и откроет приложение в браузере. Для остановки нажмите `Ctrl+C`.

На macOS можно просто дважды нажать файл `Mighty Miners.command` в Finder.

### Ручной запуск

```bash
npm install
npm run dev
```

Откройте [http://localhost:3000](http://localhost:3000).

Проверки перед демонстрацией:

```bash
npm run typecheck
npm run lint
npm run build
```

### Проверка AI-анализа договора

Главный сценарий: `/applications/new` → шаг «Загрузите документы» → загрузить договор в PDF (или выбрать «Взять демодоговор») → «Проанализировать документы». Все документы этого блока принимаются как PDF до 10 МБ. Сервер извлекает текст PDF и запускает локальный авторизованный Codex CLI через `codex exec`; браузер напрямую к CLI не обращается.

В текущем hackathon-MVP не используется прямой OpenAI API и не нужен `OPENAI_API_KEY`. Чтобы проверить настоящий анализ, установите Codex CLI по официальной инструкции OpenAI, авторизуйте его в своей учётной записи и убедитесь, что команда доступна:

```bash
codex --version
npm run dev
```

Это демонстрационный адаптер: в production тот же серверный контракт будет подключён к LLM API. Такой режим позволяет организаторам увидеть реальную работу ИИ локально, не передавая авторизацию или текст договора во frontend. В Vercel-среде Codex CLI обычно недоступен, поэтому для проверки AI-сценария используйте локальный запуск с авторизованным CLI; остальные страницы MVP продолжают работать в deployment.

Полный P0 smoke‑сценарий запускается против production‑сборки:

```bash
python3 -m pip install -r requirements-dev.txt
python3 -m playwright install chromium
npm run build
npm run start
# в другом терминале
python3 tests/smoke_home_and_wizard.py
```

Supabase для demo не нужен. Чтобы подготовить подключение, скопируйте `.env.example` в `.env.local` и заполните ключи.

## Архитектура — как объяснить жюри

### Серверный AI-анализ без API-ключа

`app/api/analyze-contract/route.ts` — единственная граница между UI и Codex CLI. Endpoint принимает только PDF до 10 МБ, проверяет расширение, MIME и сигнатуру `%PDF-`, извлекает текст на сервере, ограничивает его 24 000 символами и передаёт текст через stdin. Команда запускается массивом аргументов с `shell: false`, `--sandbox read-only` и временной директорией; пользовательский текст не попадает в аргументы командной строки. Ответ проверяется по JSON Schema, stderr не показывается пользователю, а временные файлы удаляются в `finally`.

Если PDF является сканом без текстового слоя, endpoint возвращает понятное сообщение без OCR. Документы нужны только на время анализа и в интерфейсе MVP хранятся локально в IndexedDB, чтобы их можно было открыть или скачать после загрузки.

### 1. Один источник состояния

`ApplicationProvider` в `lib/application-store.tsx` хранит заявки и предоставляет операции `addApplication`, `updateApplication` и `resetDemo`. Поэтому форма, страница покупателя и мониторинг читают одинаковые данные. В demo метаданные сделки сохраняются в `localStorage`, а сами PDF и изображения — в `IndexedDB`: статус, согласия, расчёт, оплаты, история и новые загруженные файлы переживают reload. Файлы из старой demo-версии, где были сохранены только названия, нужно загрузить заново один раз кнопкой в заявке или архиве.

Финансовые термины остаются в интерфейсе, но страница сделки объясняет их рядом с действиями. `Grace period` — это льготный период после даты оплаты: платёж уже просрочен, но возможный регресс ещё не наступил.

Реестр контроля оплат использует такой же подход через `PaymentMonitoringProvider`. Хранилища разделены по доменным задачам: создание заявки не перезаписывает бухгалтерский реестр, а адаптер `paymentMonitoringDealFromApplication` автоматически добавляет переданную P0-заявку в контроль оплат и синхронизирует её статус.

### 2. Repository pattern для Supabase

Компоненты не знают, где лежат данные. Контракт `ApplicationRepository` находится в `lib/repositories/application-repository.ts`. Сейчас его реализует `LocalApplicationRepository`; позднее можно добавить `SupabaseApplicationRepository` с теми же `list/replaceAll` и не менять UI. Singleton‑клиент уже подготовлен в `lib/supabase/client.ts`.

### 3. Форма как конечный автомат

`ApplicationWizard` хранит `step` от 1 до 4. Перед переходом выполняется валидация только текущего шага. Черновик сохраняется после каждого изменения. После отправки данные нормализуются в тип `Application`, добавляются в общий store, а черновик удаляется.

### 4. Формулы отделены от интерфейса

Компоненты только собирают ввод и показывают результат. Чистые функции находятся отдельно:

- `lib/calculate-readiness.ts` — взвешенный checklist;
- `lib/calculate-profitability.ts` — прибыль и маржинальность;
- `lib/calculate-recourse-risk.ts` — потенциальный регресс и резерв;
- `lib/calculate-deal-dates.ts` — календарные даты и сообщения.
- `lib/calculate-payment-monitoring.ts` — сроки оплаты, просрочка, дата регресса, риск, резерв и следующее действие.

Так формулы легко защищать, тестировать и переносить на backend без переписывания JSX. Пороги вынесены в `data/demo-rules.ts` и `data/demo-monitoring-rules.ts`, чтобы бизнес-правила не были «магическими числами» в компонентах.

### 5. Безопасная граница продукта

Route `/confirm/[token]` не показывает меню поставщика и даёт сети только три ответа по конкретной поставке. Повторный ответ блокируется. Расчёт отмечен как предварительный; ЭЦП, уведомления и платежи честно названы демонстрационными. Перед передачей партнёру `ConfirmationModal` блокирует кнопку, пока пользователь не даст согласие.

### 6. Статусы управляют интерфейсом

`ApplicationStatus` — строгий TypeScript union. Один статус определяет badge, активный этап timeline и доступный action‑блок. Это не набор разрозненных boolean‑флагов, поэтому нельзя одновременно показать взаимоисключающие состояния.

### 7. Ключевые demo-формулы

Для поставки 2 000 000 ₸:

```text
прибыль до факторинга = 2 000 000 − 1 720 000 − 20 000 = 260 000 ₸
стоимость факторинга = 59 178 + 9 000 + 12 000 = 80 178 ₸
прибыль после факторинга = 260 000 − 80 178 = 179 822 ₸
маржа после факторинга = 179 822 / 2 000 000 × 100 = 8,99%
финансирование = 2 000 000 × 90% = 1 800 000 ₸
потенциальный регресс = max(1 800 000 − оплачено покупателем, 0)
рекомендуемый резерв = потенциальный регресс × 20%
```

В production ставку и финальные условия возвращает финансовый партнёр.
