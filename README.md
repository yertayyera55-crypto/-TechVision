# Mighty Miners — MVP кабинета поставщика

Рабочий PWA‑frontend для локальных поставщиков: поставщик создаёт заявку по выполненной поставке, отправляет одноразовую ссылку торговой сети, получает предварительный расчёт и передаёт пакет финансовому партнёру с явным согласием.

## Запуск

Требование: установленный Node.js 20.9+.

### Самый простой способ

```bash
npm run demo
```

Команда при первом запуске сама установит зависимости, найдёт свободный порт и откроет приложение в браузере. Для остановки нажмите `Ctrl+C`.

На macOS можно просто дважды нажать файл `Mighty Miners.command` в Finder.

### Ручной запуск

```bash
npm install
npm run dev
```

Откройте [http://localhost:3000](http://localhost:3000).

Проверки перед демонстрацией:

```bash
npm run typecheck
npm run lint
npm run build
```

Полный браузерный smoke‑сценарий запускается против production‑сборки:

```bash
python3 -m pip install -r requirements-dev.txt
python3 -m playwright install chromium
npm run build
npm run start
# в другом терминале
python3 tests/smoke_home_and_wizard.py
```

Supabase для demo не нужен. Чтобы подготовить подключение, скопируйте `.env.example` в `.env.local` и заполните ключи.

## Архитектура — как объяснить жюри

### 1. Один источник состояния

`ApplicationProvider` в `lib/application-store.tsx` хранит заявки и предоставляет две операции: `addApplication` и `updateApplication`. Поэтому форма, кабинет и страница сети читают одинаковые данные. В demo они сохраняются в `localStorage`, а переход статуса сразу виден во всех разделах.

### 2. Repository pattern для Supabase

Компоненты не знают, где лежат данные. Контракт `ApplicationRepository` находится в `lib/repositories/application-repository.ts`. Сейчас его реализует `LocalApplicationRepository`; позднее можно добавить `SupabaseApplicationRepository` с теми же `list/replaceAll` и не менять UI. Singleton‑клиент уже подготовлен в `lib/supabase/client.ts`.

### 3. Форма как конечный автомат

`ApplicationWizard` хранит `step` от 1 до 4. Перед переходом выполняется валидация только текущего шага. Черновик сохраняется после каждого изменения. После отправки данные нормализуются в тип `Application`, добавляются в общий store, а черновик удаляется.

### 4. Безопасная граница продукта

Route `/confirm/[token]` не показывает меню поставщика и даёт сети только три ответа по конкретной поставке. Предварительный расчёт явно назван неофертой. Перед передачей партнёру `ConfirmationModal` блокирует кнопку, пока пользователь не поставит checkbox согласия.

### 5. Статусы управляют интерфейсом

`ApplicationStatus` — строгий TypeScript union. Один статус определяет badge, активный этап timeline и доступный action‑блок. Это не набор разрозненных boolean‑флагов, поэтому нельзя одновременно показать взаимоисключающие состояния.

### 6. Формула demo

Для поставки 2 000 000 ₸:

```text
доступно = 2 000 000 × 0,970411 = 1 940 822 ₸
стоимость = 2 000 000 − 1 940 822 = 59 178 ₸
комиссия Mighty Miners = 0 ₸
```

В production ставку и финальные условия возвращает финансовый партнёр.

## Структура

```text
app/                         routes Next.js App Router
components/                  переиспользуемый UI и сценарные компоненты
  form/                      поля, шаги и uploader
  ui/                        кнопки, badge, modal, toast, empty state
lib/
  demo-data.ts               исходные заявки отдельно от компонентов
  application-store.tsx      общее client state
  repositories/              граница источника данных
  supabase/                   подготовленный client factory
  types.ts                   доменные TypeScript-типы
tests/                       Playwright smoke-сценарий
public/manifest.json         PWA manifest
```
