# FlowFactor — AI-платформа подготовки факторинговых заявок

FlowFactor — учебный MVP для малого поставщика продуктов, который поставляет товар торговой сети и ждёт оплату 60–90 дней. Платформа помогает разобрать договор, учесть категорию товара, рассчитать подтверждённое требование, собрать универсальную заявку и сравнить предварительные варианты.

**FlowFactor не является банком или факторинговой компанией, не переводит деньги и не принимает окончательное решение о финансировании.** Компании, партнёры, критерии, комиссии, статусы и предложения в MVP синтетические.

## Быстрый запуск

Нужен Node.js 20.16+ или 22.3+.

```bash
npm run demo
```

На macOS можно открыть `Mighty Miners.command` двойным кликом. Скрипт использует уже запущенный Next.js-сервер, если он есть. Ручной запуск:

```bash
npm install
npm run dev
```

## Главный демосценарий

`Профиль компании → договор → AI-анализ → категория → расчёт требования → проверка документов → универсальная анкета → предварительный подбор партнёров → сравнение демо-предложений`

1. Нажмите «Продолжить с демо-профилем» или создайте локальный демо-кабинет.
2. Нажмите «Получить финансирование».
3. Загрузите PDF-договор либо нажмите «Запустить готовое демо». Накладная, ЭСФ, акт и документы на товар необязательны на первом шаге.
   В подготовленном договоре уже указаны порядок приёмки и условия уступки денежного требования; пустые юридические поля в демосценарии не требуются.
4. Проверьте сведения, которые AI нашёл в договоре, предварительную категорию товара и короткие фрагменты-основания.
5. Уточните принятые поставки, возвраты и удержания. Подтверждённое требование и дата оплаты рассчитываются автоматически.
6. Проверьте универсальную анкету. У каждого поля указан источник: профиль, договор, документ поставки, расчёт системы или отсутствие данных.
7. Нажмите «Подтвердить и проверить предложения» и сравните 2–3 варианта от вымышленных партнёров.

FlowFactor отмечает один вариант как наиболее подходящий по синтетическим критериям, но пользователь может выбрать любого партнёра. Особенности и дополнительные требования не блокируют кнопку выбора.

Готовый сценарий с чаем работает детерминированно, поэтому его можно показать на Vercel без AI API и Codex CLI. Для пользовательского PDF настоящий анализ выполняется локально через Codex CLI.

## Где используется AI, а где правила

AI используется только для извлечения и классификации:

- стороны и номер договора;
- предмет и категория поставки;
- сумма, срок и начало отсрочки;
- приёмка, возвраты, удержания и уступка требования;
- упомянутые в договоре документы;
- короткие фрагменты договора как основание результата.

Детерминированная логика приложения выполняет:

- проверку обязательных полей;
- категорийную матрицу документов и рисков;
- расчёт подтверждённого требования и даты оплаты;
- расчёт полноты пакета;
- сопоставление с критериями синтетических партнёров;
- расчёт и объяснение демо-предложений.

Категория влияет на документы и риски, но не создаёт автоматическую банковскую ставку или решение.

Ключевые модули:

- `app/api/analyze-contract/route.ts` — безопасный серверный запуск Codex CLI и строгая схема;
- `data/product-categories.ts` — отдельная матрица категорий, документов и рисков;
- `data/demo-partners.ts` — отдельная конфигурация критериев вымышленных партнёров;
- `lib/factoring-engine.ts` — классификация-fallback, расчёт требования, готовность и rule-based matching;
- `components/application-wizard.tsx` — пользовательский путь без ручного заполнения заявки с нуля.

## Локальный AI-анализ через Codex CLI

В MVP не используется прямой OpenAI API и не нужен `OPENAI_API_KEY`. Endpoint `/api/analyze-contract` проверяет расширение, MIME-тип, размер до 10 МБ и PDF-сигнатуру, извлекает текстовый слой и запускает установленный авторизованный Codex CLI:

```bash
codex --version
npm run dev
```

`codex exec` запускается через `child_process.spawn` без shell, в ephemeral/read-only sandbox. Текст договора передаётся через stdin, пользовательский текст не помещается в аргументы командной строки, временная папка удаляется в `finally`, а результат проходит JSON Schema и серверную валидацию. Текст договора, реквизиты и результат анализа не записываются в обычные логи. Сканированные PDF без текстового слоя получают понятную ошибку; OCR в MVP не реализован.

В полноценном продукте анализ будет работать через серверный LLM API с защищённым хранением секретов и документов. Codex CLI используется только для проверяемой локальной демонстрации хакатона: организатору нужно установить и авторизовать Codex CLI, после чего запустить проект локально. В Vercel Codex CLI обычно недоступен, поэтому там предусмотрен прозрачный подготовленный демосценарий.

## Реальное и симулированное

Реально работает: загрузка и локальное хранение PDF, извлечение текста, анализ через Codex CLI, категория, список недостающих данных и документов, вычисления, автозаполнение анкеты, rule-based matching, объяснение и сравнение.

Симулировано: партнёры и их критерии, передача заявки, скоринг, одобрение, комиссии и оферты, подписание, выплата, банковские статусы и оплата покупателя.

> Для полноценного запуска потребуются соглашения с банками и факторинговыми компаниями, их актуальные критерии и анкеты, защищённый обмен данными, юридическая проверка, согласие пользователей и интеграция со статусами заявок. В MVP эти внешние процессы представлены прозрачным синтетическим сценарием.
