### TVEP для финала

TVEP (Tech Vision Engineering Portfolio) - это ровно 3 страницы PDF, который финалисты отправляют за 24 часа до защиты:

1. Название, трек, одна боль одного пользователя и USP.
2. Архитектура: frontend, backend, базы данных, внешние API, стек и логика AI-компонента либо алгоритма.
3. Математическая/алгоритмическая модель, Cost Structure и Revenue Streams для коммерческого проекта.

Содержание этого README уже разложено так, чтобы из него можно было собрать TVEP без расхождения с кодом. Для сцены действует формат «Show, Don't Slide»: 4 минуты на питч и живую демонстрацию, 2 минуты на Q&A. Команда должна иметь короткую запись экрана как Plan B при сбое сети или внешнего сервиса.

## Задача проекта

Малому поставщику факторинг помогает получить часть выручки до наступления срока оплаты со стороны торговой сети. Перед обращением к фактору поставщику приходится повторно переносить сведения из договора, разбираться в требованиях к документам и сравнивать условия разных организаций.

FlowFactor сокращает эту подготовительную работу. Поставщик загружает договор, система извлекает сведения и собирает готовую анкету. Пользователь проверяет результат, дополняет только отсутствующие данные и сравнивает предварительные варианты.

FlowFactor не является банком или факторинговой компанией, не переводит деньги и не принимает решение о финансировании. Партнёры, критерии, ставки, статусы и предложения в текущей версии вымышлены и используются для демонстрации продуктовой логики.

## Проверяемый пользовательский сценарий

Основной путь состоит из следующих этапов:

1. Вход с готовым демо-профилем или создание локального кабинета.
2. Загрузка договора поставки в PDF.
3. Извлечение условий договора через локальный Codex CLI.
4. Проверка найденных сведений и категории товара.
5. Расчёт подтверждённого требования и даты оплаты.
6. Проверка автоматически заполненной анкеты FlowFactor.
7. Внутренняя демонстрационная проверка.
8. Сравнение предварительных предложений и выбор любого партнёра.

Пользователь начинает с договора, а не с длинной формы. После анализа в анкету переносятся:

- из профиля — название компании, БИН, отрасль, оборот, срок работы и контакты;
- из договора — покупатель, номер договора, предмет поставки, сумма, срок отсрочки, порядок оплаты, приёмки и уступки требования;
- из расчётов — подтверждённое требование, доступная сумма и предполагаемая дата оплаты.

Возле полей указан источник: «Из профиля», «Из договора», «Из документа поставки», «Рассчитано системой» или «Нужно заполнить». Если сведения не найдены, система оставляет поле пустым; модель не должна дополнять договор предположениями.

## Подготовленный демосценарий

Кнопка «Запустить готовое демо» загружает встроенный договор и детерминированный результат анализа. Этот путь работает без Codex CLI и доступен в размещённой на Vercel версии.

Данные сценария:

| Поле | Значение |
| --- | --- |
| Поставщик | ТОО «Arman Tea» |
| Покупатель | ТОО «Aspan Market» |
| Предмет поставки | Фасованный чай |
| Сумма | 10 000 000 ₸ |
| Отсрочка | 60 дней после поставки |
| Приёмка | Подписанная накладная в течение одного рабочего дня |
| Уступка требования | Разрешена после письменного уведомления покупателя; отдельное согласие не требуется |

В договоре намеренно присутствуют условия приёмки и уступки требования. Накладная, ЭСФ, акт и документы на товар на этапе демо-заявки необязательны; выбранный партнёр может запросить их позднее.

Подготовленный сценарий нужен для стабильной презентации интерфейса и расчётов. Он не выдаётся за результат обращения к внешней модели.

## Что выполняет модель

Codex CLI используется только для извлечения и предварительной классификации сведений из пользовательского PDF:

- сторон и номера договора;
- предмета и категории поставки;
- суммы, дат, срока и условий оплаты;
- порядка приёмки, возвратов и удержаний;
- условий уступки денежного требования;
- документов, прямо упомянутых в договоре;
- коротких фрагментов-оснований для найденных значений.

Модель не рассчитывает комиссию, не оценивает кредитный риск, не выбирает партнёра и не принимает решение о финансировании. Эти части либо вычисляются обычными функциями приложения, либо явно обозначены как демонстрационная симуляция.

## Что рассчитывается правилами

Приложение использует детерминированную логику для операций, результат которых должен быть воспроизводимым:

- подтверждённое требование = принятые поставки − возвраты − удержания;
- предварительно доступное финансирование ограничено 95% подтверждённого требования;
- дата оплаты = дата подтверждения поставки + срок отсрочки;
- готовность пакета учитывает наличие договора и найденные в нём условия приёмки и уступки;
- категория определяет характерные документы и риски;
- предложения рассчитываются по конфигурации вымышленных партнёров;
- один вариант получает отметку «Рекомендуем», но все варианты остаются доступными для выбора.

В категорийной матрице предусмотрены чай и кофе, мясная и охлаждённая продукция, молочная продукция, овощи и фрукты, бакалея, замороженная продукция, кондитерские изделия, напитки и резервная категория. Категория влияет на подсказки по документам и рискам, но сама по себе не означает одобрение или отказ.

## Предложения партнёров

В MVP настроены три вымышленных партнёра: Örnek Capital, Qala Finance и Bereke Flow. Для каждого заданы минимальная сумма, максимальная отсрочка, процент финансирования, демонстрационная комиссия, тип факторинга, отношение к скоропортящейся продукции и требование подтверждения приёмки.

Сопоставление показывает:

- ориентировочную сумму финансирования;
- стоимость и чистую сумму;
- тип факторинга — с регрессом или без регресса;
- причины соответствия;
- особенности, которые партнёр проверил бы дополнительно.

Несоответствие отдельному критерию не блокирует выбор. Система объясняет ограничение, а пользователь сохраняет право выбрать любого партнёра. В реальном продукте критерии и доступность должны поступать из интеграций с финансовыми организациями.

## Архитектура

```text
Браузер
  ├─ профиль, черновик и заявки: localStorage
  ├─ загруженные файлы: IndexedDB
  └─ POST /api/analyze-contract
       ├─ проверка PDF
       ├─ извлечение текстового слоя
       ├─ Codex CLI в изолированной временной папке
       ├─ проверка JSON по схеме
```

## Обработка PDF через Codex CLI

Прямой OpenAI API в MVP не используется, переменная `OPENAI_API_KEY` не нужна. Для анализа произвольного договора сервер вызывает установленный и авторизованный Codex CLI.

Обработчик `/api/analyze-contract`:

1. Принимает один договор и до трёх дополнительных PDF.
2. Проверяет расширение, MIME-тип, сигнатуру файла и лимит 10 МБ на документ.
3. Извлекает текстовый слой на сервере и ограничивает общий текст 24 000 символов.
4. Создаёт временную папку и фиксированную JSON Schema.
5. Запускает `codex exec` через `child_process.spawn` с массивом аргументов, `shell: false`, `--ephemeral` и `--sandbox read-only`.
6. Передаёт текст договора через stdin, а не через аргументы командной строки.
7. Принимает JSON из stdout и повторно проверяет типы, допустимые категории, даты и длину фрагментов.
8. Удаляет временную папку в `finally`.

Дополнительные меры:

- инструкции внутри документа считаются недоверенными данными и не должны выполняться;
- пользовательский текст не включается в командную строку;
- данные авторизации Codex не читаются и не передаются в браузер;
- текст договора, реквизиты и результат анализа не записываются в обычные серверные логи;
- stderr и stack trace не показываются пользователю;
- процесс останавливается по тайм-ауту 90 секунд;
- PDF без текстового слоя получает отдельное сообщение; OCR в MVP не реализован.

Для промышленной версии Codex CLI должен быть заменён серверным LLM API с управлением доступом, защищённым хранением секретов, аудитом и политикой удаления документов. Локальный CLI выбран для хакатона, чтобы работу анализа можно было проверить без передачи API-ключа проекту.

## Хранение данных

MVP не требует базы данных. Профиль, черновик, заявки и статусы сохраняются в `localStorage`; бинарные документы — в IndexedDB текущего браузера. Загруженный файл можно открыть или скачать со страницы документов, пока данные этого браузера не очищены.

Демо-вход не является производственной аутентификацией. Данные не синхронизируются между устройствами и браузерами. Подключение Supabase подготовлено в кодовой базе, но основной сценарий MVP от него не зависит.

## Локальный запуск

Требования:

- Node.js 20.16+ или 22.3+;
- npm;
- Codex CLI — только для анализа собственного PDF.

```bash
npm install
npm run dev
```

Приложение откроется на [http://localhost:3000](http://localhost:3000). Упрощённый запуск:

```bash
npm run demo
```

На macOS тот же сценарий запускается двойным кликом по `Mighty Miners.command`. Скрипт повторно использует уже запущенный Next.js-сервер и открывает приложение в браузере.

### Проверка подготовленного сценария

1. Открыть главную страницу.
2. Выбрать «Продолжить с демо-профилем».
3. Нажать «Получить финансирование».
4. Нажать «Запустить готовое демо».
5. Проверить извлечённые сведения, категорию и фрагменты-основания.
6. Перейти к расчёту требования и сформировать анкету.
7. Подтвердить анкету и сравнить предложения.

### Проверка анализа собственного PDF

Codex CLI должен быть установлен и авторизован в той же системе, где запущен Next.js:

```bash
codex --version
codex login
npm run dev
```

После этого на первом шаге можно загрузить PDF с текстовым слоем и нажать «Проанализировать документы». Скан без распознанного текста не обрабатывается.

## Размещение на Vercel

Размещённая версия предназначена для проверки интерфейса и подготовленного сценария. Серверная среда Vercel не содержит пользовательскую авторизацию Codex CLI, поэтому анализ произвольного PDF там недоступен. Полный путь с собственным документом проверяется локально.

## Проверка кода

```bash
npm run typecheck
npm run lint
npm run build
```

Браузерные smoke-тесты используют Playwright из `requirements-dev.txt`.

## Ограничения MVP

- нет OCR для сканированных документов;
- нет производственной регистрации и разграничения доступа;
- нет серверного долговременного хранения файлов;
- нет интеграций с банками, факторами, ЭЦП и торговыми сетями;
- нет юридической проверки договора;
- конфигурации партнёров и предложения не являются коммерческими условиями;
- расчёты служат для демонстрации интерфейса и не являются финансовой консультацией.

Для перехода к пилоту потребуются договорённости с финансовыми партнёрами, их актуальные анкеты и критерии, защищённое серверное хранилище, согласия пользователей, аудит обработки документов и интеграция со статусами заявок.


## Tech Vision 2026: расширенный обзор для жюри

Этот раздел добавляет к основному описанию проекта исследовательский контекст, точную алгоритмику, бизнес-гипотезу и карту соответствия критериям хакатона. Все утверждения ниже проверены по текущей кодовой базе; неподтверждённые командные данные помечены отдельно.

### Один пользователь - одна острая боль

Главный пользователь - владелец или финансовый менеджер небольшого локального поставщика продуктов, уже работающего с крупной торговой сетью. Пример сценария - поставщик фасованного чая.

Острая боль:

> Пока деньги поставщика заморожены из-за отсрочки платежа, ему сложно быстро получить финансирование под уже существующую поставку: он самостоятельно ищет подходящего партнёра, разбирается в разных требованиях и повторно переносит данные из договора в несколько анкет.

Это не общий тезис «у бизнеса есть кассовый разрыв». FlowFactor фокусируется на конкретном участке цепочки:

~~~text
сделал -> продал -> доставил -> дождался оплаты -> смог поставить снова
~~~

FlowFactor не отменяет отсрочку торговой сети и не выдаёт деньги. Он сокращает подготовительный путь от поставки до проверяемой универсальной заявки.

### Что такое факторинг в этом продукте

Поставщик уже поставил товар и имеет право получить оплату от покупателя в установленную дату. Лицензированный финансовый партнёр может перечислить большую часть подтверждённой суммы раньше срока, а окончательные условия и расчёты определяются факторинговой схемой.

Факторинг отличается от обычного кредита тем, что связан с конкретной поставкой и дебиторской задолженностью. FlowFactor не выдаёт абстрактный кредит на развитие бизнеса, не обещает одобрение и не утверждает, что факторинг всегда выгоднее кредита.

### Почему это Local Commerce

Local Commerce - это не только выход производителя на полку магазина. Устойчивость цифрового рынка зависит и от того, может ли поставщик продолжать производить и поставлять после первой сделки. Если деньги выпадают из оборота на 60-90 дней, выход на широкий рынок становится трудно поддерживать.

FlowFactor - нейтральная инфраструктура между локальным поставщиком, крупным покупателем и лицензированной финансовой организацией. Продукт не привязан к конкретной торговой сети.

### Ценность и новизна

FlowFactor не является обычным калькулятором и не является пустой оболочкой над LLM. Его отличает проверяемый конвейер:

~~~text
PDF
 -> проверка типа, сигнатуры и размера
 -> извлечение текстового слоя
 -> AI-извлечение в фиксированную JSON Schema
 -> серверная валидация и evidence-фрагменты
 -> указание источника каждого поля
 -> missingData для отсутствующих сведений
 -> категорийные документы и риски
 -> детерминированный расчёт требования
 -> rule-based сопоставление партнёров
 -> объяснимое сравнение предложений
~~~

Главное ценностное предложение:

> Один договор - одна универсальная заявка - несколько предварительных вариантов факторинга.

Пользователь не переносит одни и те же сведения в несколько форм, но сохраняет контроль над каждым важным полем.

### CustDev и границы исследовательского утверждения

В репозитории нет расшифровок интервью, записей звонков или обезличенных анкет. Поэтому README не выдаёт гипотезу за доказанное исследование и не придумывает цитаты.

Текущая продуктовая гипотеза основана на сценарии локального поставщика чая и требует дополнительной проверки на реальных поставщиках. Интервью должны проверить:

- повторяется ли боль у других поставщиков;
- какие документы фактически требуют финансовые организации;
- какая стоимость финансирования приемлема;
- допускают ли типовые договоры уступку требования;
- какие причины отказа встречаются чаще всего;
- готов ли поставщик загружать коммерческие документы;
- нужна ли одна универсальная анкета или разные формы для партнёров;
- насколько важны OCR, ERP-интеграции и уведомления.

В качестве гипотез, влияющих на MVP, проверяются следующие наблюдения: поставщику нужны деньги под уже выполненную поставку, повторный ввод данных раздражает, категории товара требуют разных документов, а AI-результат должен быть проверяемым и не должен заполнять пробелы догадками.

### AI: точная роль и глубина интеграции

AI используется для:

- извлечения сторон, номера договора, суммы, дат и условий оплаты;
- извлечения предмета поставки и предварительной категории;
- поиска условий приёмки, возвратов, удержаний и уступки;
- выделения документов, прямо упомянутых в договоре;
- возврата коротких evidence-фрагментов;
- формирования списка missingData и notes.

AI не рассчитывает комиссию, не выбирает партнёра, не оценивает кредитоспособность и не принимает решение о финансировании.

Глубина интеграции обеспечивается тем, что:

1. PDF проверяется до запуска модели.
2. Текст извлекается отдельным этапом через pdf-parse.
3. Используется фиксированный prompt и JSON Schema.
4. Неизвестное значение возвращается как null и попадает в missingData.
5. productCategory ограничен enum.
6. Даты проверяются форматом YYYY-MM-DD.
7. Evidence ограничен максимумом 8 фрагментов по 160 символов.
8. Результат дополнительно проверяется серверным TypeScript-кодом.
9. Расчёты и partner matching выполняются обычным rules engine.
10. Пользователь видит источник значения и может исправить категорию.

### Codex CLI как runtime-компонент

В MVP нет прямого OpenAI API и не требуется OPENAI_API_KEY. Для собственного PDF нужен установленный и авторизованный Codex CLI.

Route handler app/api/analyze-contract/route.ts:

- принимает обязательный договор и до трёх дополнительных PDF;
- ограничивает размер каждого PDF 10 МБ;
- проверяет расширение, MIME и сигнатуру PDF;
- ограничивает извлечённый общий текст 24 000 символами;
- создаёт временную директорию;
- запускает codex exec через child_process.spawn без shell;
- передаёт ephemeral и sandbox read-only;
- передаёт текст договора через stdin;
- принимает JSON из stdout;
- валидирует результат;
- завершает процесс через 90 секунд;
- удаляет временную директорию в finally;
- возвращает пользователю обобщённые ошибки без stack trace.

Содержимое договора обрабатывается как недоверенные данные: prompt запрещает выполнять инструкции, найденные внутри документа. Текст договора и результат анализа не записываются в обычные логи.

В live/Vercel-среде нет пользовательской авторизации Codex CLI, поэтому там предназначен для показа подготовленный детерминированный сценарий. Полный анализ собственного PDF проверяется локально.

### Алгоритмы и синтетические параметры

Функция lib/factoring-engine.ts рассчитывает:

~~~text
confirmedReceivable =
  max(0, acceptedSupplyAmount - returnsAmount - holdsAmount)

demoUpperLimit = round(confirmedReceivable * 0.95)

availableFinancing =
  min(demoUpperLimit,
      max(0, desiredFinancingAmount || demoUpperLimit))

dueDate = confirmationDate + max(0, paymentTermDays)
~~~

Для предложения партнёра:

~~~text
financingAmount =
  min(availableFinancing,
      round(confirmedReceivable * financingPercentage / 100))

cost =
  round(financingAmount * commissionRate * max(1, termDays) / 60)

netAmount = max(0, financingAmount - cost)
~~~

Readiness проверяет наличие договора, условия приёмки и условия уступки, считает полноту трёх базовых условий и применяет категорийные риски. Оценка ограничивается диапазоном от 25 до 95.

Для partner matching строятся caveats по сроку, минимальной сумме, подтверждению приёмки и скоропортящейся категории:

~~~text
score =
  (preferredCategory ? 25 : 0)
  + financingPercentage
  - caveatsCount * 40
  - commissionRate * 100
~~~

Партнёр с максимальным score получает «Рекомендуем». Caveats не блокируют кнопку «Выбрать предложение». Все параметры, включая 95%, комиссии и имена партнёров, синтетические.

### Категорийная логика

Категория влияет на документы, риски и демонстрационное сопоставление, но не означает автоматическое одобрение или отказ.

| Категория | Что отличается |
| --- | --- |
| Чай и кофе | Декларация соответствия; риски срока годности и маркетинговых удержаний |
| Мясо и охлаждённые продукты | Ветеринарные документы и температурный лист; холодовая цепь и повышенный риск возврата |
| Молочная продукция | Декларация соответствия; короткий срок годности и возвраты |
| Овощи и фрукты | Фитосанитарный документ при необходимости; естественная убыль и корректировки количества |
| Бакалея | Декларация соответствия; ретро-бонусы и удержания |
| Замороженная продукция | Температурный лист; холодовая цепь и споры о качестве |
| Кондитерские изделия | Декларация соответствия; сезонность и срок годности |
| Напитки | Декларация соответствия; промо- и логистические удержания |

### Хранение и безопасность

Профиль, auth demo, черновик, заявки и мониторинг хранятся в localStorage. PDF и другие бинарные файлы хранятся в IndexedDB текущего браузера. Синхронизации между устройствами нет, а demo-вход не является production-аутентификацией.

Для пилота дополнительно потребуются серверное шифрованное хранилище, разграничение доступа, tenant isolation, audit log, политика удаления, согласие пользователя, KYC/AML партнёра, юридически корректное подписание и реальные интеграции.

### CustDev: что нельзя заявлять без подтверждения

В репозитории нет интервью и числовых результатов. Поэтому нельзя писать, что боль подтверждена «десятками поставщиков», нельзя придумывать цитаты и нельзя выдавать расчёт MVP за измеренный социальный или финансовый эффект.

До защиты нужно либо добавить обезличенные факты проведённых интервью, либо прямо сказать, что текущая версия основана на первичной гипотезе и требует дальнейшей проверки.

### Альтернативы

- Ручная подача: поставщик сам ищет партнёров и многократно переносит сведения.
- Обычный кредит: не обязательно связан с конкретной дебиторской задолженностью.
- Прямой факторинг: решает финансовую задачу, но не стандартизирует подготовку пакета.
- FlowFactor: договор -> структурированные данные -> универсальная анкета -> категорийные требования -> предварительное сравнение.

FlowFactor не заменяет финансовую организацию.

### Бизнес-модель

Бизнес-модель является гипотезой. Комиссия в предложении партнёра - стоимость синтетического предложения, а не доход FlowFactor.

Возможные Revenue Streams:

- referral fee от лицензированного партнёра за успешно переданную или профинансированную заявку;
- B2B-подписка или white-label для факторинговых организаций;
- premium-возможности для поставщиков с большим числом заявок;
- корпоративная интеграция для торговых сетей.

Cost Structure:

- AI-обработка и OCR;
- серверная инфраструктура и защищённое хранение;
- безопасность, аудит и поддержка;
- партнёрские интеграции;
- юридическое сопровождение и согласия.

В MVP нет договоров с партнёрами, платежей и подтверждённой выручки.

### Масштабирование

1. Текущий MVP: один сегмент поставщиков, категория чая, локальное хранение и синтетические партнёры.
2. Пилот: несколько категорий, один настоящий финансовый партнёр, актуальная анкета и безопасная передача заявки по согласию.
3. Платформа: несколько партнёров, server storage, статусы, ERP/торговые сети, ЭЦП и регионы Казахстана.
4. Инфраструктура: API, white-label, аналитика качества заявок и автоматическое обновление критериев.

### Метрики

Метрики текущего MVP без выдуманных чисел:

- время прохождения готового сценария;
- доля автоматически заполненных полей;
- количество полей с источником;
- количество missingData;
- успешность end-to-end сценария;
- процент ошибок анализа;
- стабильность live-демо и smoke-тестов.

Метрики будущего пилота:

- время подготовки заявки;
- доля завершённых заявок;
- доля возвратов из-за документов;
- время до предложения;
- конверсия заявки в финансирование;
- объём профинансированной дебиторской задолженности;
- повторное использование сервиса.

### AI-комплаенс

Хакатон разрешает AI-инструменты для написания кода, но команда несёт полный авторский контроль:

- каждый разработчик должен понимать архитектуру и логику;
- нельзя скрывать использование AI;
- нельзя выдавать сгенерированный код за полностью ручной;
- каждый технический участник должен объяснить свой блок;
- плагиат, покупка готового кода и умышленный обман недопустимы.

Нужно различать Codex как инструмент разработки и Codex CLI как runtime-компонент анализа PDF.
