# FlowFactor

> Один договор - одна универсальная заявка - несколько предварительных вариантов факторинга.

- Команда: Mighty Miners
- Направление: Digital Economy & Future Tech
- Проблемная зона: Local Commerce
- Live MVP: https://tech-vision-81gi.vercel.app
- Репозиторий: https://github.com/yertayyera55-crypto/-TechVision

## Суть продукта

FlowFactor помогает малому поставщику подготовить заявку на факторинг по уже выполненной поставке. Пользователь загружает договор, получает структурированный паспорт сделки, проверяет найденные условия, формирует универсальную анкету и сравнивает предварительные варианты финансирования.

FlowFactor не является банком или факторинговой компанией. Он не переводит деньги, не утверждает финансирование и не заменяет юридическую или кредитную проверку. В текущем MVP финансовые партнёры, ставки, комиссии и предложения синтетические.

Короткий USP:

> Один договор - одна проверяемая заявка - несколько объяснимых вариантов факторинга.

## Пользователь и проблема

Пользователь FlowFactor - небольшой локальный поставщик, который уже поставил товар крупной торговой сети и ждёт оплату 60-90 дней.

Проблема возникает после поставки: деньги заморожены в дебиторской задолженности, а для поиска финансирования поставщику приходится вручную переносить сведения из договора, собирать документы и разбираться в разных требованиях финансовых организаций.

FlowFactor сокращает именно подготовительный путь:

~~~text
договор поставки
  -> структурированные условия сделки
  -> подтверждённое денежное требование
  -> универсальная анкета
  -> предварительное сравнение вариантов
~~~

Продукт не отменяет отсрочку платежа и не гарантирует получение денег. Он делает заявку понятной, контролируемой , проверяемой и пригодной для последующей передачи реальному финансовому партнёру.

## Основной сценарий

1. Пользователь открывает локальный демо-кабинет или продолжает с готовым профилем.
2. Загружает PDF договора поставки и, при необходимости, дополнительные документы.
3. Система извлекает текст и выделяет стороны, суммы, даты, условия оплаты, приёмки, возвратов, удержаний и уступки требования.
4. Пользователь проверяет результат и категорию товара.
5. Rules engine рассчитывает подтверждённое требование, доступную сумму и дату оплаты.
6. Система формирует универсальную анкету с источником каждого поля.
7. Пользователь проверяет анкету и получает предварительное сравнение синтетических партнёров.
8. Любой вариант можно выбрать для демонстрационного продолжения сделки.

Источники полей в анкете различаются: профиль, договор, документ поставки, расчёт системы или незаполненное значение. Если сведений нет, поле остаётся пустым и попадает в missingData.

Кнопка «Запустить готовое демо» использует встроенный PDF и фиксированный результат анализа. Этот путь не зависит от внешней модели и предназначен для стабильного показа продукта.

| Покупатель | ТОО «Aspan Market» |

| Дата поставки | 25 сентября 2026 |

- сравнение предварительных вариантов;

## AI и анализ договора

В текущей версии Codex CLI используется только для демо извлечения и предварительной классификации сведений из PDF:
(("В РЕАЛЬНОМ ПРОЕКТЕ БУДЕТ БУДЕТ API LLM , CODEX CLI ТОЛЬКО ДЛЯ ДЕМОНСТРАЦИИ"))

- поставщик, покупатель и номер договора;
- суммы, даты и срок оплаты;
- предмет и категория поставки;
- условия приёмки, возвратов и удержаний;
- условия уступки денежного требования;
- документы, упомянутые в договоре;
- короткие evidence-фрагменты;
- missingData и notes.

Модель не рассчитывает комиссию, не выбирает партнёра, не оценивает кредитоспособность и не принимает решение о финансировании. Эти действия выполняются обычным TypeScript-кодом.

Прямой OpenAI API в MVP не используется. Для анализа собственного PDF локально нужен установленный и авторизованный Codex CLI. В размещённом демо доступен подготовленный сценарий, потому что серверная среда не содержит пользовательской авторизации Codex CLI.

### Безопасность анализа

Route handler /api/analyze-contract:

- принимает договор и до трёх дополнительных PDF;
- ограничивает размер каждого файла 10 МБ;
- проверяет расширение, MIME и сигнатуру %PDF-;
- ограничивает общий извлечённый текст 24 000 символами;
- передаёт текст через stdin, а не через аргументы командной строки;
- запускает Codex через child_process.spawn с shell: false;
- использует ephemeral и sandbox read-only;
- применяет фиксированную JSON Schema;
- проверяет категории, даты, числа и длину evidence;
- завершает процесс по тайм-ауту 90 секунд;
- удаляет временную папку после обработки;
- не показывает пользователю stack trace и внутренний stderr.

Содержимое договора обрабатывается как недоверенные данные: инструкции внутри документа не должны выполняться. Сканированные PDF без текстового слоя не обрабатываются, OCR в MVP отсутствует.

## Доменная логика

### Расчёт требования

~~~text
confirmedReceivable =
  max(0, acceptedSupplyAmount - returnsAmount - holdsAmount)

demoUpperLimit =
  round(confirmedReceivable * 0.95)

availableFinancing =
  min(
    demoUpperLimit,
    max(0, desiredFinancingAmount || demoUpperLimit)
  )

dueDate =
  confirmationDate + max(0, paymentTermDays)
~~~

95% - демонстрационный верхний предел, а не обещание реального партнёра.

### Расчёт предложения

~~~text
financingAmount =
  min(
    availableFinancing,
    round(confirmedReceivable * financingPercentage / 100)
  )

cost =
  round(financingAmount * commissionRate * max(1, termDays) / 60)

netAmount =
  max(0, financingAmount - cost)
~~~

При сопоставлении учитываются срок отсрочки, минимальная сумма, необходимость подтверждения приёмки, скоропортящаяся категория и приоритетные категории партнёра. Один вариант получает статус «Рекомендуем», но все варианты остаются доступными.

### Категории товара

Категория определяет подсказки по документам, риски и демонстрационное сопоставление. Она не означает автоматическое одобрение или отказ.

| Категория | Примеры требований и рисков |
| --- | --- |
| Чай и кофе | Декларация соответствия, срок годности, маркетинговые удержания |
| Мясо и охлаждённые продукты | Ветеринарные документы, температурный лист, холодовая цепь |
| Молочная продукция | Декларация соответствия, короткий срок годности, возвраты |
| Овощи и фрукты | Фитосанитарные документы при необходимости, корректировки количества |
| Бакалея | Декларация соответствия, ретро-бонусы и удержания |
| Замороженная продукция | Температурный лист, холодовая цепь и споры о качестве |
| Кондитерские изделия | Декларация соответствия, сезонность и срок годности |
| Напитки | Декларация соответствия, промо- и логистические удержания |

## Хранение данных

Основной MVP работает без удалённой базы:

- профиль, черновики, заявки и статусы хранятся в localStorage;
- PDF и другие бинарные документы хранятся в IndexedDB;
- данные не синхронизируются между браузерами и устройствами;
- демо-вход не является production-аутентификацией;
- подключение Supabase подготовлено, но не является обязательной частью текущего сценария.

## Чего пока не хватает

В проекте пока отсутствуют:

- подтверждённые интервью с поставщиками и количественные пользовательские метрики;
- production-аутентификация, роли и multi-tenant доступ;
- серверное зашифрованное хранилище документов;
- OCR и обработка сканированных PDF;
- реальные финансовые партнёры, их API, анкеты, ставки и статусы;
- банковское API для передачи заявок и получения решений;
- ERP, бухгалтерские и EDI-интеграции с поставщиками и торговыми сетями;
- электронная подпись и юридически значимая проверка договора;
- KYC/AML и реальный кредитный скоринг;
- реальное перечисление финансирования и подтверждение оплаты;
- управляемый production LLM API вместо локального Codex CLI;
- аудит, политики хранения и удаления персональных данных.

## Возможное развитие

Если продукт подтвердит ценность на реальных поставщиках, следующая версия может включать:

1. API лицензированных банков и факторинговых организаций: актуальные условия, отправка заявки, статусы и причины отказа.
2. ERP, бухгалтерские системы, EDI и интеграции с торговыми сетями для автоматического подтверждения поставки и дебиторской задолженности.
3. Управляемую LLM-интеграцию с контролем доступа, удалением персональных данных, журналом запросов и отдельными политиками для документов.
4. OCR и мультимодальный анализ сканов с обязательной проверкой результатов пользователем.
5. Серверную архитектуру с аутентификацией, tenant isolation, объектным хранилищем и аудитом.
6. ЭЦП и юридически значимое подписание заявки, договора и документов поставки.
7. KYC/AML, юридическую верификацию и реальный workflow передачи заявки.
8. Событийный мониторинг сроков оплаты и интеграцию с подтверждениями покупателя.
9. Реальные продуктовые метрики: время подготовки заявки, доля заполненных полей, время до предложения и конверсия в финансирование.

## Локальный запуск

Требования:

- Node.js 20.16+ или 22.3+;
- npm;
- Codex CLI нужен только для анализа собственного PDF.

Запуск подготовленного сценария:

~~~bash
npm install
npm run demo
~~~

Или стандартный режим разработки:

~~~bash
npm run dev
~~~

После запуска приложение доступно на http://localhost:3000. Для анализа собственного PDF Codex CLI должен быть установлен и авторизован локально. Сканированные документы без текстового слоя не поддерживаются.
